## Vision (12-month trajectory)

_This section describes Phase 2+ ambition. Current scope (Phase 1) is internal-only.
The vision exists so Phase 1 architecture decisions preserve the path — not drive scope.
Any Phase 1 change made solely to preserve the 10x path must cost <1 hour or be deferred._

The 10x version: the coordination layer becomes the product. An AI operating system for
solo operators — the kernel is shared memory + event bus + HITL. Sales, Marketing, Ops
are default apps. Config is a single `company.yaml`. Year-2 defensibility: after 6 months,
agents have seen every lead, campaign, and ops problem for your specific company. Stateless
tools churn. This one learns.

**Platonic ideal — how it feels:**
A founder configures `company.yaml` (see schema below). A lead comes in. Sales qualifies it.
Marketing gets notified and drafts a campaign. Founder gets Slack DM with a 3-line reasoning
brief and "Approve?" One click: it sends. Monday morning: a 1-page digest lands in their
inbox, co-authored by all three agents. They spend 100% of their time on strategy only they
can do.

## Instrumentation Schema (shared contract — read before writing any agent)

Every task, regardless of agent or trigger source, writes one record:

```json
{
  "agent": "sales" | "marketing" | "ops",
  "task_id": "uuid",
  "trigger_source": "human" | "cron" | "event_bus",
  "escalated": true | false,
  "completion_time_s": number,
  "outcome": "success" | "failed" | "escalated-approved" | "escalated-rejected" | "timeout"
}
```

**Do not add new outcome values without updating:**
1. The dashboard's Trends tab logic
2. The success metric definition (Open Question #3)
3. Any existing outcome count queries

This schema is the single data contract between all agents and the dashboard.

## company.yaml Schema

```yaml
company:
  name*: string
  owner*: email
  timezone*: IANA timezone string (e.g. "America/New_York")
  departments:
    sales:
      crm*: "hubspot" | "salesforce" | "notion"
      email_platform*: "gmail" | "outlook"    # NOTE: email_platform, not email
    marketing:
      email_platform*: "sendgrid" | "mailchimp"
      social: "linkedin" | "twitter" | null
    ops:
      calendar: "google" | "outlook" | null
      comms*: "slack"
hitl:
  channel*: "slack_dm" | "email" | "slack_channel:<channel-id>"
  owner*: email
  timeout_hours: integer (default: 24)
```

_`*` = required. Startup validates config and halts with a named error on any missing
required field. Credentials (API keys, OAuth tokens) are NOT in this file — see Auth below._

**Phase 1 config update:** restart required. No hot reload. (Hot reload is YAGNI for solo
internal use — the founder can restart a local process when updating escalation thresholds.)

## Accepted Scope — Detail

### 1. company.yaml config surface

Schema defined above. Schema validator runs on startup. Engineers must implement one
validator for each required field, with a named error message per missing field. No
silent partial startup.

**Auth/credentials (separate from yaml):**
OAuth tokens live in environment variables, not in yaml. The yaml specifies provider names;
credentials are loaded at runtime from env.

| Integration | Auth type | Env var | Refresh behavior |
|---|---|---|---|
| Gmail | OAuth2 (access + refresh token) | `GMAIL_ACCESS_TOKEN`, `GMAIL_REFRESH_TOKEN` | Access tokens expire in 1 hour; agent must call Google OAuth refresh endpoint before each task that uses Gmail |
| HubSpot | OAuth2 | `HUBSPOT_ACCESS_TOKEN`, `HUBSPOT_REFRESH_TOKEN` | Access tokens expire in 6 hours; same refresh pattern as Gmail |
| SendGrid | API key (no expiry) | `SENDGRID_API_KEY` | No refresh needed; rotate manually if compromised |
| Slack | Bot token (no expiry) | `SLACK_BOT_TOKEN` | No refresh; re-authorize if token is revoked |
| Google Calendar | OAuth2 | `GCAL_ACCESS_TOKEN`, `GCAL_REFRESH_TOKEN` | 1-hour expiry, same as Gmail |

Each integration adapter must implement refresh-and-retry: if API call returns 401, refresh
token and retry once. If refresh fails, agent pauses and notifies founder via HITL channel.

### 2. Company dashboard

Single page, two tabs. Agents write outcomes to a local append-only JSONL file:
`~/.company-agent/events.jsonl`. Dashboard server (Python `http.server` or Node `serve`)
serves static HTML. Browser polls `GET /api/events?since=<timestamp>` every 5 seconds.
A minimal Node Express server (or Python Flask) serves both the static files and the
`/api/events` endpoint that reads from the JSONL file. This is NOT a pure static HTML
solution — it requires a thin local server process running alongside the agents.

**Activity tab:** chronological feed of instrumentation records — agent, task, trigger,
HITL status, outcome.

**Trends tab:** weekly aggregation of instrumentation records — escalation rate per agent
(escalated / total tasks), completion rate per agent. Plotted as line chart.

_The trends tab measures declining escalation rate as a PROXY for improving agent judgment.
This is meaningful from Week 1 regardless of SONA. SONA (deferred) is the mechanism that
might accelerate the decline; the metric exists independent of SONA._

### 3. Voice of the agent

Applies to: any task that triggers HITL escalation (escalated = true). Does NOT apply to
routine/cron tasks.

Before HITL notification fires:
1. Agent identifies target entity key (e.g. lead email for Sales, contact name for Mktg)
2. Issues HNSW search with **N=3** to `shared.*` and `{dept}.*` namespaces, query = entity key
3. Composes 3-line brief from retrieved memory hits (or cold-start fallback if 0 hits)
4. Sends Slack DM or email: brief + "Reply 'approve' or 'reject' within {timeout_hours}h."

**Phase 1 approval mechanism:** keyword reply. Agent polls the HITL channel for the
configured timeout period. On "approve": execute. On "reject": cancel task, record
`outcome: escalated-rejected`, log rejection in agent's memory for future learning (so
it can inform the Voice of the Agent next time a similar task arises). On timeout: cancel,
record `outcome: timeout`, notify founder.

**Cold-start (empty memory):** brief says "No prior memory for [entity]. Proceeding with
CRM data only." Do not fabricate context.

### 4. Weekly ops digest

CronCreate fires every Monday at 08:00 per `company.yaml timezone`.

**Mechanism: Ops agent reads memory directly (no sub-agent spawning).**
Ops does NOT spawn sub-agents. It queries each department's memory namespace directly:
- `sales.*` events from the prior 7 days → Sales section
- `marketing.*` events from the prior 7 days → Marketing section
- `ops.*` events from the prior 7 days → Ops section

This avoids the orchestration complexity of runtime agent spawning. Ops reads the shared
memory store — which all agents have already written to — and assembles the digest from
existing data.

Cold-start: if any namespace has < 3 records, that section reads:
"Not enough data yet — check back next week."

**On email send failure** (rate limit, invalid credentials, network error):
log `outcome: failed` in instrumentation, notify founder via Slack DM that digest failed
to send, write assembled digest text to `ops.digest_queue` memory key as fallback.

## HITL Workflow — Complete

```
Task triggers HITL?
    │
    ├─── No (routine task) ──► execute directly ──► log instrumentation ──► DONE
    │
    └─── Yes
          │
          ├─ N=3 HNSW search for entity context
          ├─ Compose 3-line Voice brief
          │
          ├─ Send to HITL channel (Slack DM / email)
          │   "Brief. Action. Reply 'approve' or 'reject' within Nh."
          │
          ├─ Poll for reply (every 60s)
          │
          ├─── "approve" received ──► execute ──► log escalated-approved ──► DONE
          │
          ├─── "reject" received ──► cancel ──► log escalated-rejected
          │                           └── write rejection note to agent memory ──► DONE
          │
          └─── timeout (no reply in N hours) ──► cancel ──► log timeout
                          └── notify founder: "Task auto-cancelled: [task description]
                               Original trigger data retained — re-queue manually if needed."
                               ──► DONE
```

## Ops Agent — Behavior Spec (non-digest days)

The Ops agent has three trigger types:

| Trigger | Frequency | Action | Escalates? |
|---|---|---|---|
| Cron (daily 07:00) | Daily | Read all three departments' memory for prior 24h, write `ops.daily_summary` | Never (internal) |
| Cross-agent event (scheduling) | As-needed | Update calendar based on Sales/Mktg events (e.g. follow-up meeting scheduled by Sales) | If external calendar invite to person not previously invited |
| Cron (Monday 08:00) | Weekly | Weekly digest (described above) | Never |
| Human-initiated | On demand | Execute named admin task (e.g. "generate Q2 pipeline report") | If any external action required |

The daily summary is written to shared memory, not emailed. The founder reads it via the
dashboard Activity tab or can ask the Ops agent to surface it.

## Coordination Layer: Event Bus

Week-0 spike validates AgentDB pub/sub. The diagram below shows the post-resolution state.
**The spike result determines which path is implemented — only one path exists in production.**

**Path A (spike passes — AgentDB native pub/sub):**
```
Sales agent writes outcome ──► AgentDB publishes event ──► Marketing subscribes ──► triggered
```
Real-time push. ~50ms latency.

**Path B (spike fails — polling fallback):**
```
Sales agent writes to event_queue.* memory key ──► Marketing polls every 5s ──► triggered
```
Eventually consistent. ~5s latency for Sales→Marketing handoff. Acceptable for all
currently-defined use cases (lead qualification, campaign triggers, Ops scheduling updates).

**Spike overrun contingency:** If the Week-0 spike takes >1 week (e.g., AgentDB behavior
is undocumented and requires debugging), default to Path B immediately. Do not let the spike
block agent development beyond Week 1. Path B is fully functional for Phase 1 use cases.

## AgentDB Namespace Convention + Known Collision Risks

```
shared.*      — cross-agent readable/writable (e.g. shared.company_context)
sales.*       — Sales agent primary namespace
marketing.*   — Marketing agent primary namespace
ops.*         — Ops agent primary namespace
event_queue.* — pub/sub events (Path B only)
```

**Known concurrent-write risks:**

| Key | Written by | Risk | Resolution |
|---|---|---|---|
| `shared.lead_<id>` | Sales (qualify) + Marketing (update status after campaign) | Sales writes score, Marketing updates engagement — same key | Last-write-wins acceptable: these are different fields if serialized as JSON. Document exact fields per agent. |
| `ops.daily_summary` | Ops (daily cron) only | Single writer — no collision | N/A |
| `event_queue.latest` | All agents (on any outcome) | Two agents completing simultaneously → one event lost | Accept for Phase 1. Fix in Phase 1b: use append-only event log with sequence numbers. |

## HNSW Retrieval Scope

HNSW retrieval (N=5 for task context; N=3 for Voice of the Agent) applies to:
- **Entity-context tasks:** lead qualification, outreach drafting, campaign personalization
- **Voice of the Agent:** before any HITL escalation

HNSW retrieval does NOT apply to:
- **Scheduled/routine tasks:** daily summary, weekly digest, pipeline reports
- These tasks query memory namespaces directly (not HNSW) for structured data

This distinction matters for latency and AgentDB load. Do not add HNSW retrieval to
routine cron tasks without explicit justification.

## Error Handling

From design doc failure table (HITL timeout, event bus failure, shared memory unreachable,
invalid agent output) — applies to all agents. Additional cases from accepted scope:

| Scenario | Behavior |
|---|---|
| company.yaml missing required field | Hard stop on startup; named error + fix instructions |
| OAuth token expired | Agent refresh-and-retry once; if refresh fails, pause + notify founder |
| Weekly digest email send fails | Log outcome: failed; Slack DM to founder; write digest to ops.digest_queue |
| Dashboard server not running | Agents continue normally; dashboard is a read-only consumer |
| HNSW cold-start (empty memory) | Skip HNSW; log memory_hit: 0; proceed with task data only |
| company.yaml changed on disk | Agents re-read config at task start (file-read, not watch). Restart required for changes to take effect immediately. |

The design doc says expand autonomy after 2-week validation ramp but defines no pass/fail.
Definition: an agent passes validation when ALL of the following hold over any rolling 14-day window:
1. HITL approval rate ≥ 90% (founder approves ≥ 9 of every 10 escalated tasks)
2. Zero HITL timeouts (all escalated tasks resolved within timeout_hours)
3. Zero incorrect actions (no escalated-rejected tasks where rejection was due to agent error, not founder preference)
Once an agent passes, it may have ONE trigger type un-gated from HITL (must be explicitly chosen).
All other trigger types remain HITL-gated until each passes validation independently.

### Deferred Features (Phase 1b / v1.1)

- **Replay mode** — agent evolution visibility. Requires rich Phase 1b memory + SONA.
- **Integration healthcheck** — startup verification that all integrations are live. Phase 1 v1.1.
- **Slack Block Kit HITL** — keyword "approve/reject" is functional for Phase 1. Block Kit is Phase 2 polish.
- **event_queue sequence numbers** — Phase 1 accepts last-write-wins collision risk. Fix in Phase 1b.

## Dream State Delta

**Current state (manual, today):**

A solo founder handles every department interaction manually. Lead comes in → manually qualify → manually draft email → manually send. Marketing campaigns → manually plan → manually execute. Weekly ops → manually check pipeline → manually write digest. Founder spends ~60% of their time on execution, ~40% on strategy.

**Phase 1 dream state (end of internal validation):**

- Lead comes in → Sales agent qualifies automatically → Slack DM with 3-line brief + "Approve?" → founder one-tap approve → email sends. Founder time per lead: <30 seconds.
- Pipeline event → Marketing agent drafts campaign stub → HITL → approved → scheduled. Founder time per campaign: <5 minutes.
- Monday morning → Ops digest arrives in inbox automatically. Founder reads in 2 minutes, knows the week.
- Dashboard open → Activity tab shows every agent action, HITL outcome, completion. No mystery what agents did.
- After 2-week validation → one trigger type per agent runs without HITL. Trust earned incrementally.

**What changes for the founder:**
- Execution work: ~60% → ~10% (HITL review replaces execution)
- Visibility: zero (email threads, CRM manual) → complete (dashboard + weekly digest)
- Trust in agents: zero → measurable (escalation rate curve on Trends tab)

## Error & Rescue Registry

Complete error handling — all agents, all integration types:

| Error | Agent | Behavior | Recovery |
|---|---|---|---|
| company.yaml missing required field | All (startup) | Hard stop; named error + fix instructions | User fixes yaml, restarts |
| OAuth token expired (Gmail, HubSpot, GCal) | Sales / Mktg / Ops | Refresh-and-retry once | If refresh fails: pause task, notify founder via HITL channel |
| API key invalid (SendGrid, Slack) | Sales / Mktg / Ops | Log `outcome: failed`; notify founder via HITL channel | User rotates key in env vars |
| HITL timeout (no reply in N hours) | Any escalated task | Cancel task, log `outcome: timeout`, notify founder | Founder re-queues manually |
| HITL channel unavailable (Slack offline) | Any escalated task | Fall back to email HITL (if configured); else log `outcome: failed` + write to `{dept}.hitl_queue` | Founder reads queue on Slack restore |
| AgentDB write fails | All | Write to local buffer file (`~/.company-agent/buffer.jsonl`); retry on reconnect | Auto-retry; notify founder if buffer > 100 entries |
| AgentDB read fails (HNSW unavailable) | Voice of Agent | Skip HNSW; log `memory_hit: 0`; proceed with task data only | Non-fatal; agents degrade gracefully |
| Event bus unavailable (Path A) | Sales → Mktg handoff | Fall back to Path B polling automatically | Spike result determines Path at build time |
| Weekly digest email send fails | Ops | Log `outcome: failed`; Slack DM to founder; write digest to `ops.digest_queue` | Founder retrieves from queue on demand |
| Webhook signature invalid (HMAC mismatch) | Sales (CRM webhook) | Reject with 401; log rejected request | Investigate CRM webhook config |
| Webhook payload missing required fields | Sales (CRM webhook) | Reject with 400; log missing fields | Investigate CRM payload format |
| Agent process crash mid-HITL | Any | On restart: check `*.hitl_pending.*` keys; re-send notification with "Re-sent:" prefix | Automatic recovery — see TODO-2 |
| Config file changed on disk | All | Per-task re-read (Option B, pending TODO-5 resolution) | No action required |

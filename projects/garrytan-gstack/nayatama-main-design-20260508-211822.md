## Problem Statement

A single person running a company currently handles every function manually or relies on
fragmented SaaS tools that don't talk to each other. The goal: three specialized AI agents
(Sales, Marketing, Operations) that are each excellent standalone, can coordinate across
departments via shared memory, escalate high-stakes decisions to a human, and self-evolve
from past context — so one person can run what previously required a full team.

## Demand Evidence

Currently at hypothesis stage — not validated with external users. Internal use at the
builder's own company is the validation phase. Success metric for external deployment TBD
(see P4 below).

## Status Quo

The problem is in concept phase — no existing solution handles cross-department agent
coordination. Existing workarounds: per-department SaaS AI tools (HubSpot AI, Jasper,
etc.) that don't share context. No coordination layer exists in the market.

## Target User & Narrowest Wedge

**Phase 1 (internal):** Builder's own company. All three departments simultaneously.
**Phase 2 (external):** Other companies, after internal stability is proven.

**Validation prototype (pre-Approach B, separate from main execution):**
One working Sales → Marketing handoff: Sales qualifies a lead, writes to shared memory,
Marketing agent receives trigger and sends follow-up campaign. This single handoff is the
proof-of-concept for the shared memory + event bus architecture before all three agents
are built. Estimated time: 48 hours. This is a prerequisite validation step, not part of
Approach B's parallel build phase.

## Constraints

- Must build all three agents simultaneously (user's explicit requirement)
- Each agent must be excellent standalone before coordination is the selling point
- **HITL invariant:** Human-in-the-loop check runs before execution regardless of trigger
  source (human-initiated OR cron/background). No action bypasses HITL if it meets the
  escalation threshold. This is an architectural invariant, not a conditional.
- Escalation threshold (concrete):
  - Any outbound email to a contact not previously emailed by this agent
  - Any spend or budget commitment >$0 (all financial actions escalate until threshold raised)
  - Any contract creation, modification, or signature
  - Any action affecting a human outside the company
  - Routine tasks (internal reports, scheduling, CRM updates, draft creation) do NOT escalate
- Background/scheduled tasks must run without human trigger (HITL check still applies above)
- gstack is the infrastructure foundation (not from scratch)
- Internal use before external deployment
- Integration decisions (CRM, email platform, social, Slack) must be finalized before
  starting Approach B execution; effort estimate of 2-week coordination layer is
  +unknown until integrations are confirmed (blocking dependency)

## Approaches Considered

### Approach A: Sequential Excellence
Each agent built to production quality before the next. Sales first (6 weeks), then
Marketing, then Ops. Shared memory architecture from day 1.
- Effort: M (human: ~4-5 months total / CC: ~3 weeks)
- Risk: Low
- Pros: Each agent ships and validates before next; clear feedback loop
- Cons: 4+ months before full vision exists; first agent lacks cross-department context

### Approach B: Parallel Foundation (CHOSEN)
Build coordination layer first (shared memory, event bus, HITL framework) in ~3-4 weeks
(revised from 2 weeks — event bus and HITL checkpoint require custom builds, 3-5 days
each). Then all three agents built simultaneously on top of it. Each starts "thin"
(minimal viable capability) but coordination-aware from day 1. Phase 1b adds depth.
- Effort: L (human: ~4 months total / CC: ~3-4 weeks)
- Risk: Medium
- Pros: All three exist early; coordination never retrofitted; handoffs testable from week 4
- Cons: Individual agents thin early; upfront architecture design + 1 spike task required

**Phase milestones:**
- Week 0-1: Event bus spike — validate AgentDB can support pub/sub (if not, build standalone)
- Week 1-4: Coordination layer (shared memory namespace, event bus, HITL checkpoint system)
- Week 5+: All three agents simultaneously, thin milestone first, then depth phase
- "Thin" milestone per agent: Sales = accepts webhook + writes lead score to memory + HITL
  trigger for outbound. Marketing = subscribes to Sales events + writes campaign draft to
  memory. Ops = cron trigger + writes internal report to memory.
- "Depth phase" (Phase 1b): each agent reaches excellent standalone quality; SONA/self-evolution
  added; agents do NOT go external until depth phase complete.

### Approach A: Sequential Excellence — rejected.
One agent to production quality before the next. Slower path to full vision; rejected because
the user requires all three simultaneously.

### Approach C: Memory-First — rejected as primary approach, elements incorporated.
Manual context capture before agents; core insight (P5/memory is the moat) absorbed into
Approach B. Full memory-first as the primary approach rejected due to 4-week manual discipline
requirement and delayed agent delivery.

## Recommended Approach

**Approach B: Parallel Foundation**

Rationale: The user wants all three departments simultaneously AND insists on coordination.
Building three excellent-standalone agents and retrofitting coordination afterward is
painful and expensive. Building the coordination layer first means each agent slot fills
in over time and works with the others from day 1. The shared memory namespace, event
bus, and HITL checkpoint system become the durable infrastructure all three agents plug into.

**P1 compatibility note:** Approach B starts agents "thin" (minimal capability), which
appears to conflict with P1 (individual excellence as the foundation). These are compatible
if execution treats them as two phases: (1) coordination layer + thin agents → (2) parallel
depth-building for each agent's standalone quality. The agents do NOT go external until each
is excellent standalone. "Thin but coordination-aware" at week 3 is the starting point for
the depth-building phase, not the shipped product.

### Architecture Sketch

HITL is a conditional branch off each agent action (not a universal downstream layer).
Every action above the escalation threshold triggers a HITL check before execution,
regardless of whether the trigger was human-initiated or cron/background.

```
Trigger (human or cron)
        │
        ▼
┌───────────────────────────────────────────────────────────┐
│           Agent Action                                     │
│  ┌─────────────┐  ┌───────────────┐  ┌───────────────┐  │
│  │ Sales Agent │  │  Mktg Agent   │  │  Ops Agent    │  │
│  │ lead qual   │  │  content      │  │  scheduling   │  │
│  │ pipeline    │  │  campaigns    │  │  reporting    │  │
│  │ outreach    │  │  social       │  │  admin        │  │
│  └──────┬──────┘  └──────┬────────┘  └──────┬────────┘  │
```

## Success Criteria

- Each agent handles its core function WITHOUT human initiation for at least 20 tasks/week
- At least one cross-department handoff working end-to-end (Sales → Marketing)
- Human-in-the-loop engaged for high-stakes decisions, skipped for routine tasks
- Agents improve measurably (fewer escalations, faster task completion) over 4-week window

**Week-1 instrumentation (required to measure the above):**
From day 1, log every task to a metrics store:
`{agent, task_id, trigger_source, escalated: bool, completion_time_s, outcome}`
where `outcome` ∈ `{success, failed, escalated-approved, escalated-rejected, timeout}`.
Without this baseline, the 4-week improvement delta cannot be calculated.
This is a build prerequisite, not an afterthought.

## Failure Handling

| Scenario                              | Behavior                                                         |
|---------------------------------------|------------------------------------------------------------------|
| HITL approval times out (background)  | Cancel task, log as `outcome: timeout`, notify via Slack/email  |
| HITL notification channel             | Slack message (default) or email; configurable. Required before any background agents run. |
| Event bus delivery failure            | Retry 3× with 30s backoff; dead-letter to `ops_alert` memory key if all fail |
| Shared memory unreachable             | Agent pauses, logs error, does not execute. No silent fallback.  |
| Agent produces invalid output         | Log as `outcome: failed`, surface to human review queue         |
| Unintended action executed (no undo)  | Undo log: out of scope Phase 1. Accepted risk. Mitigation: HITL threshold catches most. |

**HITL notification for async/background tasks (required):**
A background agent that triggers an escalation at 3am needs a channel to reach the human.
This must be configured before background agents are enabled. Default: Slack DM to owner.
Fallback: email. Without this, the HITL invariant is unenforceable for cron-triggered tasks.

## Event Bus Interface (sketch)

The event bus is a custom pub/sub system built on AgentDB. Minimum interface:

```typescript
// Publish (agent emits event after completing a task)
publish(event: {
  source_agent: "sales" | "marketing" | "ops",
  event_type: string,           // e.g. "lead_qualified"
  payload: Record<string, any>, // e.g. { lead_id, score, tags }
  memory_key: string            // key in shared AgentDB where full context lives
})

// Subscribe (agent registers interest in events from other agents)
subscribe(filter: {
  source_agent?: string,
  event_type?: string
}, handler: (event) => void)

// Week-0 spike: validate AgentDB supports this pattern natively. (1-2 days)
// If spike passes: event bus is a thin wrapper on AgentDB. Timeline unchanged.
// If spike fails: implement as a polling loop on shared `event_queue` memory key.
//   Fallback adds ~3 days. Reliability note: polling is eventually consistent
//   (agent polls every 5s); acceptable for Sales→Marketing latency, review if
//   Ops requires real-time triggers. Total coordination layer estimate with
//   fallback: 4-5 weeks instead of 3-4 weeks.
```

Payload schema for Sales → Marketing handoff:
`{ lead_id: string, score: number, tags: string[], next_step: string, memory_key: string }`

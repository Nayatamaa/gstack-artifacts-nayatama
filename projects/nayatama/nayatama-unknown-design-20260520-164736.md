## Problem Statement

Nexor AI manages back-office exception decisions (bookkeeping claims, transaction
categorisation, CoSec filings) manually, one case at a time. No institutional memory
exists — every new exception is decided from scratch, even when precedent exists.
Layer B (the regulatory change ingestion layer that parses SSM/LHDN advisories into
structured records) is ingested manually with no structured parsing or approval workflow.

Core hypothesis to prove: an AI assistant (Nex) that receives past decisions as context
produces better suggestions than a context-free assistant. The human approval gate is
the quality control mechanism.

---

## Demand Evidence

Internal demand confirmed. Kenneth (reviewer) handles these exceptions today. The
prototype contains a live pending exception — Aiman's Saturday Grab receipt — which
is a real case, not synthetic. Validation criterion: Nex references Seed 1 (prior Grab
receipt decision, see Seed Data section) in its PRECEDENT field when resolving this case.

## Status Quo

Manual decision-making per exception, no memory across cases. Regulatory changes (SSM/LHDN)
read manually and applied ad hoc. No structured audit trail beyond email threads.

## Target User & Narrowest Wedge

**Phase 1 (internal):** Kenneth and the Nexor AI team. The single pending exception

**Phase 2 (external):** Malaysian SMEs with SSM/LHDN compliance workflows. Requires

## Constraints

- Single HTML file — no build tools, opened via local server (see CORS note)
- No hardcoded API keys in the HTML file
- Nexor brand palette strictly applied — no deviations (see Nexor Brand Palette section)
- Nex voice: under 150 words per response, PRECEDENT > SUGGESTION > RATIONALE > NOVELTY FLAG
- Human gate required before any state change (no auto-propagation)
- Audit trail append-only — INSERT only, never DELETE or UPDATE enforced in code
- OpenAI API key hidden behind Cloudflare Worker (not visible in browser source)
- Supabase anon key in source config block (Supabase security model — designed to be
  public-facing; RLS protects data in production, see RLS Note)
- Prototype database must contain no real PII or financial data until RLS is configured
- Internet access required at runtime (Cloudflare Worker, Supabase, Google Fonts CDN)

## Nex Context Serialization

How past decisions are passed to the API on every "Ask Nex" call:

```js
async function callNex(situation, exceptionCategory) {
  // Fetch most recent 50 decisions as context (practical ceiling for GPT-4o context window)
  const decisions = await supabaseSelect(
    'decision_records',
    'order=created_at.desc&limit=50'
  )

  const messages = [
    {
      role: 'system',
      content: NEX_SYSTEM_PROMPT  // the full prompt defined in "Nex System Prompt" section
    },
    {
      role: 'user',
      // Seeds + any logged decisions injected as a serialized list
      content: `Past decisions for reference (most recent first):\n${JSON.stringify(decisions, null, 2)}`
    },
    {
      role: 'user',
      content: `New exception to resolve:\nCategory: ${exceptionCategory}\nSituation: ${situation}`
    }
  ]

  const res = await fetch(AI_CONFIG.endpoint, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ model: AI_CONFIG.model, messages })
  })
  const json = await res.json()
  return json.choices?.[0]?.message?.content ?? ''
}
```

Context ceiling note: `limit=50` is sufficient for Phase 1 (a handful of exceptions).
If the prototype runs for months, add a category filter: `exception_category=eq.${exceptionCategory}`
to keep context focused. Do not implement this now -- add only if context window errors appear.

## Schema Form Field Specification

Tab 3 two-column layout. Left column = form. Right column = Nex panel.

When triggered by "Resolve" on a pending exception, the form pre-fills from the queue record:

| Field | Type | Pre-filled from queue? | Required to submit? |
|-------|------|------------------------|---------------------|
| Exception Category | Dropdown (bookkeeping_claim / tx_categorisation / cosec_filing) | Yes | Yes |
| Situation | Textarea | Yes, editable | Yes |
| Decision | Textarea | No -- blank | Yes |
| Rationale | Textarea | No -- blank | Yes |
| Source Documents | Text input | No -- blank | Yes |

Submit button: disabled (grey, cursor not-allowed) until all 5 fields have non-empty content.
Check on every keystroke/change event -- do not wait for submit click.

"Accept" from Nex populates Decision and Rationale from Nex's SUGGESTION and RATIONALE
fields. User can edit before submitting. Source Documents must still be filled manually.

On submit: UPDATE the existing pending exception record in decision_records (do not insert
a new row). Use supabaseUpdate with the pending exception's ID to fill in decision,
rationale, source_documents, reviewer ('Kenneth'), ai_used, ai_suggestion, ai_accepted.
Then write audit event, navigate back to Tab 2, refresh queue and Brain Overview counters.

The Tab 2 queue distinguishes resolved vs pending by checking whether decision IS NULL.
After update, the record's decision field is non-null, so it moves from pending to resolved
display on next queue refresh.

## Cloudflare Worker Script

Deploy in Cloudflare dashboard (Workers and Pages > Create > paste > Save).
Add OPENAI_API_KEY as encrypted secret in Settings > Variables.
Optional: add CLAUDE_API_KEY if Claude API swap is needed later.

```js
export default {
  async fetch(request, env) {
    const cors = {
      'Access-Control-Allow-Origin': '*',
      'Access-Control-Allow-Methods': 'POST, OPTIONS',
      'Access-Control-Allow-Headers': 'Content-Type',
    }
    if (request.method === 'OPTIONS') return new Response(null, { headers: cors })
    if (request.method !== 'POST')
      return new Response('Method not allowed', { status: 405, headers: cors })

    const path = new URL(request.url).pathname
    const body = await request.text()

    let upstream, headers
    if (path === '/chat-claude') {
      // Claude API swap: this route + model name in AI_CONFIG + x-api-key header
      upstream = 'https://api.anthropic.com/v1/messages'
      headers  = {
        'x-api-key': env.CLAUDE_API_KEY,
        'anthropic-version': '2023-06-01',
        'Content-Type': 'application/json',
      }
    } else {
      upstream = 'https://api.openai.com/v1/chat/completions'
      headers  = {
        'Authorization': `Bearer ${env.OPENAI_API_KEY}`,
        'Content-Type': 'application/json',
      }
    }

    const res  = await fetch(upstream, { method: 'POST', headers, body })
    const data = await res.text()
    return new Response(data, {
      status:  res.status,
      headers: { ...cors, 'Content-Type': 'application/json' },
    })
  },
}
```

## Supabase Schema

Run this SQL in the Supabase SQL Editor before starting the app.

```sql
-- Layer A: Decision records (institutional memory)
-- decision/rationale/source_documents/reviewer are nullable to support pending exceptions
-- (records inserted from the queue before Kenneth has resolved them).
-- Schema Form enforces non-empty at the UI level before submit.
create table decision_records (
  id                 uuid default gen_random_uuid() primary key,
  exception_category text not null
    check (exception_category in ('bookkeeping_claim','tx_categorisation','cosec_filing')),
  situation          text not null check (length(situation) > 0),
  decision           text,   -- NULL for pending exceptions; required before submit in UI
  rationale          text,   -- NULL for pending exceptions
  source_documents   text,   -- NULL for pending exceptions
  reviewer           text,   -- NULL for pending exceptions; DB default 'Kenneth' when omitted
  ai_used            boolean default false,
  ai_suggestion      text,
  ai_accepted        boolean,
  is_seed            boolean default false,
  created_at         timestamptz default now()
);

-- Layer B: Regulatory changes
create table regulatory_changes (
  id                  uuid default gen_random_uuid() primary key,
  regulator           text not null check (regulator in ('SSM','LHDN')),
  source_label        text not null,
  raw_text            text not null,
  parsed_output       jsonb,
  status              text default 'pending'
    check (status in ('pending','approved','rejected')),
  approved_by         text,
  approved_at         timestamptz,
  rejection_rationale text,
  created_at          timestamptz default now()
);

-- Audit trail (append-only -- never UPDATE or DELETE this table)
create table audit_events (
  id          uuid default gen_random_uuid() primary key,
  actor       text not null,
  actor_type  text not null check (actor_type in ('human','ai','system')),
  event_type  text not null,
  subject     text not null,
  ai_involved boolean default false,
  payload     jsonb,
  created_at  timestamptz default now()
);
```

## Supabase API Pattern (corrected v2 shape)

supabaseInsert returns { data, error } so the atomicity guard works correctly
regardless of Supabase REST response shape.

```js
async function supabaseSelect(table, filters = '') {
  const res = await fetch(`${SUPABASE_URL}/rest/v1/${table}?${filters}`, {
    headers: {
      'apikey': SUPABASE_ANON_KEY,
      'Authorization': `Bearer ${SUPABASE_ANON_KEY}`
    }
  })
  return res.json()
}

async function supabaseInsert(table, data) {
  const res = await fetch(`${SUPABASE_URL}/rest/v1/${table}`, {
    method: 'POST',
    headers: {
      'apikey': SUPABASE_ANON_KEY,
      'Authorization': `Bearer ${SUPABASE_ANON_KEY}`,
      'Content-Type': 'application/json',
      'Prefer': 'return=representation'
    },
    body: JSON.stringify(data)
  })
  const json = await res.json()
  // Supabase returns array on success, {code,message} object on error
  return { data: res.ok ? json : null, error: res.ok ? null : json }
}

async function supabaseUpdate(table, id, data) {
  const res = await fetch(`${SUPABASE_URL}/rest/v1/${table}?id=eq.${id}`, {
    method: 'PATCH',
    headers: {
      'apikey': SUPABASE_ANON_KEY,
      'Authorization': `Bearer ${SUPABASE_ANON_KEY}`,
      'Content-Type': 'application/json'
    },
    body: JSON.stringify(data)
  })
  return res.ok
}
```

Write-atomicity pattern (prior learning applied -- write before publish):

```js
// Only write audit event after decision_records INSERT confirms success
const { data, error } = await supabaseInsert('decision_records', payload)
if (!error) {
  await supabaseInsert('audit_events', auditPayload)
}
```

## RLS Note

The Supabase anon key is designed to be public-facing (Supabase uses RLS for access
control, not key secrecy). For this prototype:
- No RLS configured -- full table access via anon key
- Do not store real PII, real financial figures, or real client names in the prototype DB
- Use realistic-looking but fictional data

Minimum RLS before any external-facing use:
```sql
alter table decision_records enable row level security;
alter table regulatory_changes enable row level security;
alter table audit_events enable row level security;

create policy "anon_read" on decision_records for select to anon using (true);
create policy "auth_write" on decision_records for insert to authenticated with check (true);
-- Repeat pattern for regulatory_changes and audit_events
```

## Implementation Sequence

Build in this order. Each step testable before the next.

1. Cloudflare Worker -- deploy, set secret, confirm URL responds to test POST
2. Supabase -- create project, run schema SQL, confirm 3 tables exist
3. HTML shell -- 5-tab nav, brand palette, fonts, all tabs empty but navigable
4. Tab 2 + Tab 3 -- seed insertion check, Exception Queue, Schema Form + Nex Panel
   (core hypothesis test -- get here before anything else)
5. Tab 1 + Tab 4 + Tab 5 -- Brain Overview counters, Layer B Feed, Audit Trail
6. Cross-tab state:
   - Tab 2 "Resolve" click: store the full pending exception record (as JSON) in a
